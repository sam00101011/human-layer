# Provider Setup

This is the opinionated setup guide for Human Layer using:
- Vercel
- Neon
- Upstash
- Sentry
- PostHog

It is written to match the current repo and env surface.

## Chosen stack

- Web app: Vercel
- Database: Neon Postgres
- Redis: Upstash Redis
- Error tracking: Sentry
- Product analytics: PostHog Cloud

Use placeholder URLs until you choose final production and staging hostnames.

## Setup order

1. Create Neon projects.
2. Create Upstash Redis databases.
3. Create Sentry projects.
4. Create PostHog project.
5. Create World ID production credentials.
6. Create the Vercel web project.
7. Add environment variables.
8. Deploy web.
9. Run migrations.
10. Build the extension against the hosted web URL.

## 1. Neon

Create:
- one production project
- one staging project

For each project:
- choose a primary region close to your first users
- enable automatic backups
- create a database named `human_layer`

Collect:
- direct Postgres connection string for `DATABASE_URL`

Recommended naming:
- `human-layer-prod`
- `human-layer-staging`

Use:
- production Neon connection string in Vercel production envs
- staging Neon connection string in Vercel preview/staging envs

Notes:
- keep the direct connection string for migrations
- if you later add serverless pooling separately, keep runtime and migration URLs distinct

## 2. Upstash Redis

Create:
- one production Redis database
- one staging Redis database

Collect:
- standard Redis connection string for `REDIS_URL`

Recommended naming:
- `human-layer-prod`
- `human-layer-staging`

Use Redis first for:
- rate limits
- short-lived lookup cache
- queue backing once worker jobs land

## 3. Sentry

Create projects:
- `human-layer-web`
- `human-layer-worker` later when the worker exists

Collect:
- server DSN for `SENTRY_DSN`
- browser DSN for `NEXT_PUBLIC_SENTRY_DSN`

Recommended settings:
- enable source map upload later in CI
- enable performance monitoring
- enable release tracking once deploy flow stabilizes

## 4. PostHog Cloud

Create one PostHog project for Human Layer.

Collect:
- project API key for `NEXT_PUBLIC_POSTHOG_KEY`
- PostHog host for `NEXT_PUBLIC_POSTHOG_HOST`

Use PostHog for:
- verification funnel
- extension handoff success
- overlay open and retry interactions
- page/profile clickthrough events

## 5. World ID

Create the production World ID app and request-signing credentials.

Collect:
- `NEXT_PUBLIC_WORLD_ID_APP_ID`
- `WORLD_ID_APP_ID`
- `WORLD_ID_RP_ID`
- `WORLD_ID_RP_PRIVATE_KEY`
- `WORLD_ID_VERIFY_URL`

Set:
- `WORLD_ID_MODE=remote`
- `WORLD_ID_ENVIRONMENT=production`
- `NEXT_PUBLIC_WORLD_ID_ACTION=human-layer-v1`
- `WORLD_ID_ACTION=human-layer-v1`
- `NEXT_PUBLIC_WORLD_ID_SIGNAL=human-layer-v1`
- `WORLD_ID_SIGNAL=human-layer-v1`

## 6. Vercel

Create one Vercel project for the web app.

Recommended project shape:
- framework: Next.js
- root directory: `apps/web`
- node version: `22`

If Vercel does not automatically detect the workspace correctly:
- install command: `corepack pnpm install --frozen-lockfile`
- build command: `corepack pnpm --filter @human-layer/web build`

Recommended environments:
- Production
- Preview

Use preview deployments for staging-like verification even before you choose the final public domain.

## 7. Environment variable mapping

### Provided by Neon

- `DATABASE_URL`

### Provided by Upstash

- `REDIS_URL`

### Provided by World ID

- `NEXT_PUBLIC_WORLD_ID_APP_ID`
- `WORLD_ID_APP_ID`
- `WORLD_ID_RP_ID`
- `WORLD_ID_RP_PRIVATE_KEY`
- `WORLD_ID_VERIFY_URL`

### Provided by Sentry

- `SENTRY_DSN`
- `NEXT_PUBLIC_SENTRY_DSN`

### Provided by PostHog

- `NEXT_PUBLIC_POSTHOG_KEY`
- `NEXT_PUBLIC_POSTHOG_HOST`

### Generated by you

- `SESSION_SECRET`
- `EXTENSION_TOKEN_SECRET`

Generate both as long random values and store them only in Vercel.

### Base URL placeholders

Set these even before the final domain exists:
- `APP_BASE_URL=https://<current-web-deployment-base-url>`
- `API_BASE_URL=https://<current-api-base-url>`

If the API still lives on the same Next app, both can point to the same base URL for now.

## 9. First deploy

After envs are set:

1. Deploy `apps/web` on Vercel.
2. Confirm the home page loads.
3. Confirm `/api/pages/lookup?url=https%3A%2F%2Fgithub.com%2Fvercel%2Fnext.js` returns JSON.
4. Confirm `/verify` loads with the expected World ID mode.
5. Run database migrations against production.

Do not run seed data in production unless you intentionally want demo content visible.

## 10. Extension build config

For production extension builds set:
- `WXT_APP_URL=https://<current-web-deployment-base-url>`
- `APP_URL=https://<current-web-deployment-base-url>`

Use the same values in:
- CI release workflow
- local production build verification

## Recommended staging shape

Before a final public launch, keep:
- a separate Vercel preview or staging environment
- a separate Neon database
- a separate Upstash Redis instance
- separate Sentry environment tagging

Do not share staging and production databases.

## Operational checks after setup

After the web app is deployed:

- verify lookup route works for a GitHub repo URL
- verify verify-flow page loads
- verify extension handoff works
- verify request-id logging appears on lookup failures
- verify Sentry receives a test event
- verify PostHog receives a test client event

## What to postpone

Do not add yet:
- custom API service split
- worker autoscaling complexity
- Kubernetes
- dedicated search service

Those become worthwhile later, not before the hosted app and extension loop are stable.
